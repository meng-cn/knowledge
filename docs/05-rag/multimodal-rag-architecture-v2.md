## 第1章 概述与演进

### 1.1 为什么需要多模态 RAG
- 纯文本 RAG 的局限性：表格、图表、图片、视频、音频信息的缺失
- 企业知识库中非结构化多模态数据的规模（估算：80%+ 为多模态）
- 多模态 RAG 的典型场景：医疗影像问答、合同条款检索、产品目录搜索

### 1.2 多模态 RAG 与传统 RAG 对比
- **架构图**：传统 RAG vs 多模态 RAG 架构对比图
- **对比表**：数据源类型、嵌入空间、检索策略、响应格式、延迟要求
- **核心差异**：嵌入空间统一化、检索结果异构性、响应合成复杂度

### 1.3 多模态 RAG 核心技术栈
- **架构图**：多模态 RAG 全栈技术栈分层图（数据层 → 嵌入层 → 检索层 → 融合层 → 生成层 → 输出层）

### 1.4 架构演进：Text-Centric → Native Multimodal
- **演进路线图**：
  ```
  Phase 1: Text-Centric（先 OCR/转文字 → 文本检索 → LLM 生成）
     │
     ├── 优势：简单、成熟、低成本
     ├── 劣势：丢失视觉信息、OCR 错误、表格还原失败
     └── 代表方案：Tesseract + BM25 + GPT-4
     │
     ▼
  Phase 2: Hybrid（部分模态原生 + 部分转文字）
     │
     ├── 优势：平衡效果与成本
     └── 代表方案：CLIP 图像检索 + 文本 BM25
     │
     ▼
  Phase 3: Native Multimodal（所有模态原生统一）
     │
     ├── 优势：保留全部视觉/音频信息、语义对齐更准确
     ├── 劣势：计算成本高、工程复杂
     └── 代表方案：ColPali、GPT-4o 原生多模态
  ```
- 选型决策：何时走 Text-Centric，何时走向 Native Multimodal

## 第8章 多模态 RAG 架构设计

### 8.1 架构设计模式总览
- **架构图**：四种经典架构模式
  1. **文本中心化（Text-Centric）**
  2. **原生多模态（Native Multimodal）**
  3. **混合架构**
  4. **检索-生成一体化**

### 8.2 Text-Centric 架构深度解析 ⭐
- **核心思路**：所有模态先转文本 → 统一文本检索 → LLM 生成
- **数据流图**：图像→OCR→文本 / PDF→解析→文本→检索→LLM
- **优势**：成熟、简单、成本低、可复用现有 RAG 管线
- **劣势**：丢失视觉信息、OCR 错误不可逆、复杂表格还原失败
- **OCR 幻觉陷阱** ⭐：OCR 把代码里的 `;` 识别成了 `:` 导致 RAG 失败 —— 真实坑位
- **适用场景**：文本密集、图像/表格较简单、成本敏感

### 8.3 Native Multimodal 架构深度解析 ⭐
- **核心思路**：各模态原生统一编码 → 跨模态检索 → 多模态 LLM 生成
- **数据流图**：图像/文本/视频 → 多模态嵌入 → 跨模态向量检索 → GPT-4o/Qwen-VL
- **优势**：保留全部视觉/音频信息、语义对齐更准确、不依赖 OCR
- **劣势**：计算成本高、工程复杂、需要多模态 LLM
- **ColPali 方案**：直接对文档页面进行视觉特征编码
- **适用场景**：复杂文档（含流程图、类图、复杂表格）、高精度要求

### 8.4 混合架构
- **核心思路**：简单内容走 Text-Centric，复杂内容走 Native Multimodal
- 路由策略：根据内容复杂度动态选择架构路径
- **架构图**：混合架构的决策分流图

### 8.5 架构选型决策树 ⭐
- **决策树**：根据业务场景、模态复杂度、成本预算选择架构
